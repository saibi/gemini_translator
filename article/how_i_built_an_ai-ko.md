## 시작: 하나의 대화

이 모든 것은 단 하나의 Claude Code 세션에서 시작되었습니다.

터미널을 열고 나의 상황을 설명하기 시작했습니다: 우리 제품이 무엇인지, 어떤 기능을 하는지, 타겟 고객이 누구인지, 어떤 경쟁사가 있는지, 우리의 가치 제안이 무엇인지. Ahrefs MCP 서버를 연결해서 Claude가 현재 키워드 순위를 가져오고 우리의 현재 위치를 파악할 수 있게 했습니다. 그런 다음 요청했습니다: _시장 진출(go-to-market) 전략을 만들어줘._

Claude는 90일 콘텐츠 플랜을 내놓았습니다 — SEO 핵심 주제(pillar)에 매핑된 블로그 포스트 주제, 소셜 미디어 배포 주기, 플랫폼 우선순위, 주간 마일스톤까지. 막연한 "아이디어 목록" 문서가 아니었습니다. 구체적이고 순서가 정해진 실행 계획이었습니다.

그때부터 하나씩 실행해 나갔습니다. "첫 번째 블로그 포스트를 작성해줘." "소셜 미디어 트래커를 설정해줘." "콘텐츠 캘린더를 만들어줘." 대화할 때마다 시스템에 새로운 레이어가 추가되었습니다. 며칠 안에 작동하는 마케팅 운영 체계를 갖추게 되었습니다 — 처음부터 전체 아키텍처를 설계해서가 아니라, 다음에 필요한 것을 계속 Claude에게 만들어 달라고 요청했기 때문입니다.

## 사고방식 전환: 프롬프트 작성이 아니라 직원 채용

모든 것을 명확하게 만든 첫 번째 전환점: 프롬프트에 대한 생각을 멈추고 직무 기술서(job description)에 대한 생각을 시작하는 것입니다.

누군가를 채용할 때, 다음을 제공합니다:

- 직함과 담당 범위 (무엇을 소유하는지)
- 스타일 가이드 (어떻게 행동해야 하는지)
- 회사 문서 (필요한 컨텍스트)
- 적절한 도구 (사용할 수 있는 것들)
- 일정 (언제 무엇을 해야 하는지)

제가 구축한 것이 정확히 이것입니다 — 직원 핸드북 대신 마크다운 파일로 만들었을 뿐입니다.

## 팀 구조

다섯 개의 에이전트가 있습니다:

![](https://miro.medium.com/v2/resize:fit:1400/1*PSVBT9NDeOAzGGrvDpLW0A.png)
AI 마케팅 팀 에이전트들

각 에이전트는 `.claude/agents/` 디렉토리 안의 단일 마크다운 파일입니다. 그게 전부입니다. Claude Code는 이 파일들을 읽고, CMO가 작업을 위임할 때 전문화된 서브에이전트(subagent)를 생성합니다.

## 실험적 기능: 에이전트 팀

이 모든 것은 **에이전트 팀(agent teams)**이라는 실험적 Claude Code 기능을 기반으로 합니다. 기본 Claude Code 경험이 아닙니다 — 하나의 에이전트가 다른 에이전트들을 서브프로세스로 생성하고 조율할 수 있게 해주는 옵트인(opt-in) 기능입니다.

크론 잡(cron job)이 실행될 때 실제로 일어나는 일:

1. Claude Code가 시작되고 CMO를 활성화합니다
2. CMO는 주간 플랜을 읽고, 현재 시간대를 파악하고, 무엇을 해야 하는지 결정합니다
3. 각 작업에 대해 CMO는 Claude Code의 `Agent` 도구를 사용해 **전문 서브에이전트를 생성**합니다
4. 해당 서브에이전트는 자체 격리된 컨텍스트 윈도우, 자체 시스템 프롬프트(`.claude/agents/` 파일에서), 자체 도구 세트를 가집니다
5. 실행 후 결과를 보고하고 종료합니다
6. CMO는 다음 작업으로 넘어갑니다

이것이 강력한 이유는 전문가들이 **병렬로** 실행될 수 있기 때문입니다. 오늘 슬롯에 블로그 포스트 게시와 소셜 미디어 인게이지먼트 세션이 모두 필요하다면, CMO는 콘텐츠 작성자와 소셜 미디어 마케터를 동시에 생성할 수 있습니다. 컨텍스트를 공유하거나 서로 방해하지 않습니다 — 각각 독립적으로, 목적에 맞게 설계된 시스템 프롬프트로 실행됩니다.

에이전트들은 몇 가지 메커니즘을 통해 조율합니다:

- **태스크 시스템** — CMO가 태스크를 생성하고(`TaskCreate`), 전문가에게 할당하고, 완료를 추적합니다(`TaskUpdate`). 각 에이전트는 시작할 때 태스크 목록을 확인하고 완료 시 태스크를 완료로 표시합니다.
- **파일 시스템** — 에이전트들은 파일을 통해 상태를 공유합니다. 소셜 미디어 마케터는 트래커 CSV에 데이터를 추가하고, 성과 검토자(Performance Reviewer)가 읽습니다. 단순하지만 효과적입니다.
- `**SendMessage**` — 에이전트들은 완료하거나 막혔을 때 CMO에게 직접 메시지를 보낼 수 있습니다. CMO는 이를 새로운 대화 턴으로 받습니다.

또 다른 핵심 개념은 **에이전트 메모리**입니다. 각 에이전트에는 세션 간에 유지되는 영구 메모리 디렉토리(`.claude/agent-memory/<agent-name>/`)가 있습니다. CMO는 승인된 전략, 진행된 캠페인, 효과적이었던 메시지를 기억합니다. 소셜 미디어 마케터는 이미 답글을 달았던 계정을 기억합니다. 이것이 시스템을 상태 없는(stateless) 스크립트가 아닌 조직 지식을 갖춘 팀처럼 느끼게 만드는 것입니다.

이 모든 것은 내 쪽에서 특별한 인프라가 필요 없습니다 — 마크다운 파일이 있는 `.claude/agents/` 폴더와 Claude Code의 오케스트레이션만 있으면 됩니다.

## 에이전트 파일의 실제 모습

에이전트 파일은 두 부분으로 구성됩니다: 프런트매터(frontmatter)와 시스템 프롬프트.

프런트매터(상단의 YAML)는 Claude Code 동작을 설정합니다:

```yaml
name: content-writer
description: "블로그 포스트를 작성할 때 이 에이전트를 사용..."
model: sonnet
color: green
memory: project
```

`description` 필드는 CMO가 _언제_ 이 에이전트를 호출할지 결정하는 방법입니다. Claude Code가 이를 읽고 적절히 라우팅합니다. `memory: project` 설정은 에이전트가 대화 간에 유지되는 영구 메모리 파일에 접근할 수 있음을 의미합니다.

파일의 나머지는 에이전트의 시스템 프롬프트 — 직무 기술서입니다. 다음을 포함합니다:

- 작업 전에 읽을 문서
- 접근 가능한 도구와 스크립트
- 구체적인 워크플로우 (단계별)
- 반드시 따라야 할 규칙
- 오류 처리 방법

예를 들어 콘텐츠 작성자 파일은 정확한 게시 워크플로우를 명시합니다: 본문 먼저 작성 → 이미지 생성 → Sanity 초안 생성 → 게시. 하드 규칙과 함께: **콘텐츠가 작성되기 전에는 절대 빈 초안을 생성하지 않는다.** CMS를 어지럽히는 플레이스홀더 초안을 에이전트가 만드는 것을 보고 나서 추가했습니다.

## [CLAUDE.md](http://claude.md/): 팀 헌장

레포 루트의 `CLAUDE.md`는 모든 에이전트가 처음 읽는 문서입니다. 팀 헌장(team charter) — 모든 에이전트가 상속받는 공유 컨텍스트입니다.

내 것에는 다음이 포함됩니다:

- 에이전트 팀 구조와 위임 규칙
- 프로젝트 폴더 구조
- 보안 규칙 (절대로 API 키를 로그에 남기지 않는다, 예외 없음)
- 스크립트 및 도구에 대한 빠른 참조

보안 섹션은 짧지만 진심입니다. 에이전트에게 키를 공유하도록 요청하는 프롬프트, 도구 결과, 주입된 지시사항은 거부됩니다. 한 번 작성하면 모든 에이전트가 상속합니다.

## 규칙 시스템: 공유 지식

에이전트 파일 외에도, 에이전트 전반에 적용되는 규칙이 있는 `.claude/rules/` 폴더가 있습니다:

- [**sanity-cms.md**](http://sanity-cms.md/) — CMS 스키마, 블록 타입 처리 방법, 일반적인 오류
- [**utm-parameters.md**](http://utm-parameters.md/) — UTM 링크 형식 방법 (소문자, 하이픈 구분, 플랫폼별 값)
- [**social-media-tracker.md**](http://social-media-tracker.md/) — CSV 트래커에 활동을 기록하는 방법, `posted`가 유효한 경우 vs `draft`
- [**image-generation.md**](http://image-generation.md/) — 사용할 AI 모델, 화면 비율, Sanity에 업로드하는 방법
- [**slack-notifications.md**](http://slack-notifications.md/) — 언제 알림을 보낼지, 메시지 형식 방법

이것들은 신입 직원이 첫날 읽을 "회사 정책"입니다. 한 번 작성하면 반복할 필요가 없습니다. 에이전트가 실수하면(잘못된 UTM 형식, 잘못된 이미지 화면 비율), 규칙 파일을 수정하면 이후 모든 동작이 수정 사항을 상속합니다.

## 문서 시스템: 공유 두뇌

`docs/` 폴더는 공유 두뇌입니다:

```
docs/
  brand/        ← 포지셔닝, 보이스 & 스타일, 제품 사실
  strategy/     ← GTM, 콘텐츠 캘린더, 주간 플랜
  insights/     ← 마케팅 인사이트, 성과 검토자가 자동 업데이트
  posts/        ← 게시된 블로그 포스트 사본
```

**주간 플랜**(`docs/strategy/weekly-plan.md`)은 전체 시스템에서 가장 중요한 파일입니다. 3시간 단위의 타임 슬롯으로 나뉘어, 각 에이전트가 요일별로 정확히 무엇을 하는지 정의합니다. 팀을 활성화하면 CMO는 현재 시간을 확인하고 올바른 슬롯을 찾아 해당 플랜에 따라 작업합니다. 에이전트들은 앞당겨 작업하지 않습니다 — 현재 슬롯과 이전 두 슬롯에서 완료되지 않은 태스크만 처리합니다.

이 단일 파일이, 매 세션 시작 시 컨텍스트를 계속 다시 설명해야 했던 것을 대체했습니다.

## 커스텀 스크립트: 플랫폼 접근

일부 플랫폼에는 공식 Claude Code 통합이 없어서, Claude에게 경량 Node.js 스크립트를 작성하게 했습니다. **이것들을 직접 작성하지 않았습니다** — 필요한 것을 설명하기만 했습니다("OAuth2를 사용해서 Reddit에 댓글을 달 수 있는 스크립트가 필요해")그러면 Claude가 코드를 작성하고, 테스트하고, 반복 개선했습니다.

결과:

- `**scripts/reddit.js**` — Reddit OAuth2 클라이언트. 소셜 미디어 에이전트가 스레드 검색, 컨텍스트 읽기, 댓글 달기, 받은 편지함 확인에 사용합니다.
- `**scripts/hn.js**` — Hacker News 헤드리스 클라이언트. HN 에이전트가 링크 제출, 댓글, 카르마 확인에 사용합니다.
- `**scripts/generate-image.js**` — Google Gemini를 호출해서 이미지 생성 (대표 이미지, 인라인 삽화).
- `**scripts/upload-image.js**` — 생성된 이미지를 Sanity CDN에 업로드하고 에셋 ID를 반환합니다.
- `**scripts/slack-message.js**` — 로그를 읽지 않고도 확인할 수 있도록 내 `#team-ai-marketing` Slack 채널에 알림을 게시합니다.

에이전트들은 실행 가능한 스크립트가 정확히 무엇인지 알고 있습니다. 그것도 규칙 파일에 있습니다 — 에이전트별 허용 목록(whitelist)으로.

## MCP 서버: 진정한 통합

적절한 API가 있는 플랫폼의 경우, 스크립트 대신 MCP(Model Context Protocol) 서버를 사용합니다. 에이전트들은 이를 네이티브 도구로 호출합니다:

- **Sanity CMS** — 포스트 생성, 콘텐츠 패치, 초안 게시, 이미지 관리
- **X/Twitter** — 트윗 게시, 대화 검색, 사용자 팔로우, 미디어 업로드
- **Slack** — 채널 읽기, 메시지 보내기
- **Ahrefs** — SEO 지표, 키워드 순위, 경쟁사 분석 가져오기

MCP 서버는 이러한 작업들을 셸 명령이 아닌 일급 작업(first-class operation)처럼 느끼게 합니다. 콘텐츠 작성자가 Sanity 문서를 "패치"하는 것은 파일을 편집하는 것과 같습니다.

## 피드백 루프

시스템은 핸드오프 체인을 통해 스스로 운영됩니다:

```
소셜 미디어 마케터 → 모든 것을 트래커 CSV에 기록
성과 검토자 → CSV 읽기, 일일 리포트 생성
CMO → 리포트 읽기, 전략 문서 업데이트
콘텐츠 작성자 → 전략 문서 읽기, 다음 콘텐츠 작성
소셜 미디어 마케터 → 게시된 콘텐츠 배포
… 반복
```

성과 검토자는 인사이트(무슨 메시지가 반향을 일으켰는지, 무엇이 무시되었는지, 무엇이 인게이지먼트를 유발했는지)로 `docs/insights/marketing-insights.md`를 자동 업데이트합니다. 다른 모든 에이전트는 작업 전에 이 파일을 읽습니다. 따라서 Twitter 답글 형식이 효과를 잃으면, 그 인사이트가 하루 안에 시스템 전반에 전파됩니다.

CMO도 세션 간에 영구 메모리를 가집니다. 내가 승인한 것, 거절한 것, 캠페인 결과, 이전 주의 패턴 — 모두 축적됩니다.

## Slack을 통한 현황 파악

완전 자율은 훌륭합니다 — 무언가 잘못될 때까지는. 내 해결책: 에이전트들이 모든 것을 `#team-ai-marketing`이라는 비공개 Slack 채널에 보고하는 것입니다.

모든 세션은 CMO가 일일 플랜을 게시하는 것으로 시작됩니다. 다른 것을 건드리기 전에, 주간 플랜을 읽고, 타임 슬롯을 확인하고, 구조화된 브리핑을 게시합니다:

- 오늘 날짜와 활성화된 타임 슬롯
- 오늘 각 에이전트가 무엇을 하는지
- 이전 슬롯에서 넘어온 태스크
- 마감일 (블로그 게시일, 캠페인 런칭)

커피를 마시기 전에 그 메시지가 Slack에 올라옵니다. 팀이 무엇을 하려는지 정확히 알 수 있습니다.

그때부터 각 에이전트는 Slack 스크립트의 `--username` 플래그를 사용해 모든 메시지에서 이름으로 스스로를 식별합니다 — `CMO`, `Content Writer`, `Social Media Marketer`:

```bash
node scripts/slack-message.js \\
  --channel C0AG3DDLZQS \\
  --username "Content Writer" \\
  --text "블로그 포스트 게시됨: <https://runbear.io/posts/inbox-zero-is-dead>"
```

블로그 포스트가 게시되면 콘텐츠 작성자가 URL을 게시합니다. 소셜 미디어 세션 후에는 마케터가 플랫폼별로 그룹화된 인게이지먼트한 모든 링크의 스레드를 게시합니다 — 각각을 탭해서 검토할 수 있도록 Slack 링크 형식으로. 성과 검토자가 실행된 후에는 CMO가 당일 지표가 포함된 요약을 게시합니다.

실패도 보고됩니다. 콘텐츠 작성자가 Sanity 스키마 오류에 부딪히거나 이미지 생성 API가 실패하면, 무엇이 잘못되었고 무엇을 시도했는지 게시합니다. 로그 파일을 뒤지지 않고 문제를 파악할 수 있습니다.

결과는 팀 스탠드업처럼 읽히는 Slack 채널입니다 — 내가 자는 동안 진행되는.

## 매시간 실행하기

잘 설계된 에이전트 팀을 갖추는 것은 한 가지입니다. 실제로 여러분 없이 실행하게 하는 것은 또 다른 문제입니다.

저는 항상 켜져 있는 전용 머신으로 침실에 **Mac Mini M1**을 사용합니다. 조용하고, 전력 소비가 최소화되며, 노트북과 달리 실행 중에 절전 모드로 전환되지 않습니다. 한 번 설정하고 나서는 물리적으로 건드린 적이 없습니다.

원격 접근에는 [**Jump Desktop**](https://jumpdesktop.com/)을 사용합니다. 팀이 막히거나 세션을 확인하고 싶을 때, 폰이나 노트북에서 Jump를 열면 몇 초 안에 Mac Mini의 화면을 보고 있습니다. 여러 대륙에 걸쳐 안정적으로 작동해왔습니다.

자동화 자체는 간단한 crontab 항목입니다. 매시간 cron이 깨어나서 프로젝트에 대해 헤드리스 모드로 Claude Code를 실행합니다:

```bash
0 * * * * cd /Users/.../ai-marketing-team && \
  claude --dangerously-skip-permissions \
  -p "당신은 CMO입니다. 현재 시간을 확인하고, 주간 플랜을 읽고, 현재 타임 슬롯의 태스크를 실행하세요." \
  >> /tmp/claude-marketing.log 2>&1
```

몇 가지 주목할 점:

- `**--dangerously-skip-permissions**`는 이것을 완전 자율로 만드는 플래그입니다. 이 플래그 없이는 Claude Code가 모든 파일 편집, 스크립트 실행, API 호출에 대해 일시 중지하고 승인을 기다립니다. 이 플래그를 사용하면 에이전트가 감독 없이 끝까지 실행됩니다. 모든 에이전트 동작을 이미 검토하고 신뢰하는 프로젝트에서만 이 플래그를 사용하세요.

**로그 파일**(`/tmp/claude-marketing.log`)은 모든 것이 기록되는 곳입니다. 실행된 내용을 검토하고 싶을 때 Jump Desktop을 통해 확인합니다. 에이전트들도 완료 시 `#team-ai-marketing`에 Slack 알림을 보내므로, 보통 로그 대신 Slack을 읽으면 됩니다.

**타임 슬롯 시스템이 여기서 효과를 발휘합니다.** 모든 에이전트가 어떤 슬롯에 있는지 알고 그 슬롯에서 무엇을 해야 하는지 정확히 알기 때문에, 시간당 cron은 많이 말할 필요가 없습니다. "현재 타임 슬롯의 태스크를 실행해"로 충분합니다 — 주간 플랜 파일이 나머지를 처리합니다.

블로그 포스트와 소셜 세션이 얼마나 많이 실행되느냐에 따라, 전체 비용은 Claude API 사용료로 하루 약 20~50달러입니다. 풀타임 마케팅 직원 대비 반올림 오차 수준입니다.

## 일상: CEO가 되기

시스템이 가동되고 나서, 내 역할은 실행이 아닌 관리가 되었습니다. 그리고 그 관리는 놀랍도록 가볍습니다.

내 상호작용은 대부분 이런 모습입니다: Claude Code를 열고, "소셜 미디어 마케터가 어떤 플랫폼에서도 em 대시를 절대 사용하지 않도록 업데이트해줘"와 같이 입력하면, Claude가 에이전트 파일을 편집합니다. 또는 "HN 계정이 섀도우밴을 받았어 — 세션당 댓글을 하나로 제한하는 규칙을 추가해줘." Claude가 규칙을 업데이트하면 다음 세션에서 동작이 바뀝니다.

성과 검토자가 특정 트윗 형식이 잘 안 된다고 보고하면, Claude에게 **"소셜 마케터가 X를 그만하고 Y를 대신 시도하도록 업데이트해줘"**라고 말합니다. 콘텐츠 작성자가 실수를 한다고 알아채면, "Z를 방지하는 하드 규칙을 추가해줘"라고 말합니다. 이 각각은 시스템을 영구적으로 개선하는 30초짜리 대화입니다.

에이전트 파일을 텍스트 에디터에서 한 번도 열어본 적이 없습니다. 모든 변경은 Claude를 통해 이루어집니다. 원하는 것을 일반 영어로 설명하면, 마크다운 파일이 업데이트됩니다.

가장 가까운 비유: 완벽한 기억력을 가지고 있고 정책 업데이트를 절대 잊지 않는 직원들과 Slack으로 원격 팀을 관리하는 것과 같습니다.

## 놀라웠던 점

**가장 어려운 부분은 코드가 아니었습니다 — 코드가 없었습니다.** 충분히 정확한 명세(specification)를 작성하는 것이었습니다. 에이전트들은 예상치 못한 방식으로 모호함을 해석합니다. 첫 주에 소셜 미디어 에이전트가 Reddit 스레드에 이중 게시를 했습니다. "중복 댓글 없음"이 충분히 구체적이지 않았기 때문입니다. Claude에게 예시와 함께 규칙을 다시 작성하게 했더니 문제가 해결되었습니다.

**규칙은 복리 효과가 있습니다.** 각 규칙을 추가하면 그것을 읽는 모든 에이전트의 동작이 개선됩니다. 명확한 규칙을 작성하는 것의 투자 대비 수익(ROI)은 개별 출력을 수정하는 것에 비해 엄청납니다.

**타임 슬롯 시스템이 모든 것을 바꿨습니다.** 없었다면 CMO가 너무 많이 하거나(아직 승인하지 않은 작업을 소진) 너무 적게 했을 것입니다(하나의 태스크 후 중단). 슬롯당 명시적인 태스크가 있는 3시간 타임 슬롯이 예측 가능한 실행 윈도우를 제공했습니다.

## 파일 수

위의 모든 것이 대략 15~20개의 파일에 들어갑니다:

- 에이전트 파일 5개
- [CLAUDE.md](http://claude.md/) 1개
- 규칙 파일 6개
- 여러 전략 문서
- 커스텀 스크립트 4개

프레임워크 없음. 특별한 인프라 없음. 좋은 문서를 갖춘 잘 정리된 레포만 있으면 됩니다.

누군가를 채용하고, 그들이 실제로 무엇을 해야 하는지 이해할 수 있을 만큼 명확하게 책임을 작성해 본 적이 있다면 — 이것을 어떻게 구축하는지 이미 알고 있는 것입니다.
